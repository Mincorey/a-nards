## 0. Что сделано хорошо (чтобы не сломать при правках)

- **Сервер авторитетен**: ходы онлайн-партии идут только через Edge Functions
  (`play-move`, `roll-dice`, `start-game`), которые заново валидируют легальность
  тем же движком. Клиент не может писать в `games`/`moves` напрямую (нет
  INSERT/UPDATE-политик — только `service_role` в обход RLS).
- **RLS продумана**: helper видимости `can_see_table` вынесен в схему `private`
  (не экспонируется как RPC), `SECURITY DEFINER`-функции с `set search_path=''`,
  `finalize_game` доступна только `service_role`, триггерная `handle_new_user`
  с отзывом EXECUTE.
- **Секреты в порядке**: на фронте только `anon`-ключ (публичный по дизайну),
  `service_role` в коде фронта отсутствует, `.env.local` в `.gitignore`.
- **Движок коротких нард корректен** (стартовая расстановка 15+15, бой блота,
  бар, вынос с правилом перебора, максимальное использование кубиков и правило
  «играть больший кубик»).
- Есть юнит-тесты (движок, правила, онлайн, username, взаимодействие доски).

## 1. 🟠 High — исправить до публикации

### H1. Самопринятие заявки в друзья (авторизация)
> ✅ **Исправлено и ПРИМЕНЕНО** (2026-07-03) к проекту `wiwfzmwzkjmluhnrrcnh`: миграция `0008_friendship_accept_guard.sql` (триггер `trg_friendships_guard_update` активен, security-advisor чист).
`friendships` политика `fr_update` разрешает UPDATE и `requester_id`, и
`addressee_id`. Функция `acceptRequest(id)` делает `update({status:'accepted'})`
по id. Значит **отправитель заявки может сам принять свою исходящую заявку** и
добавить человека в друзья без его согласия (UI это не показывает, но API/RLS
позволяют). Влияние ограничено списком друзей, но это дыра в авторизации.
**Фикс:** принятие заявки разрешать только адресату. Варианты:
- триггер `before update`, запрещающий переход `pending→accepted`, если
  `auth.uid() <> addressee_id`;
- либо отдельная `SECURITY DEFINER` RPC `accept_friend(id)` с проверкой адресата,
  а прямой UPDATE статуса запретить политикой.

### H2. Предсказуемость кубиков в онлайне (честность/анти-чит)
> ✅ **Исправлено и ПРИМЕНЕНО** (2026-07-03): броски переведены на криптостойкий `secureRng` (Edge `roll-dice`/`start-game` v3, задеплоены), колонка `games.seed` удалена (миграция `0009`). Предсказать броски по строке `games` больше нельзя.
`roll-dice` использует детерминированный `mulberry32(game.seed ^ ply*const)`.
Поле `games.seed` лежит в строке `games`, а политика `games_select` даёт
участникам читать **всю строку**, включая `seed`. Зная `seed` и `ply`, любой из
игроков может заранее вычислить все будущие броски (и свои, и соперника).
**Фикс:** не отдавать `seed` клиенту. Варианты:
- вынести `seed` в отдельную таблицу `game_secrets` без SELECT-политики (доступ
  только `service_role`);
- либо колонночные привилегии/представление `games_public` без `seed`, а клиент
  читать из него;
- либо бросать криптослучайно на сервере и хранить только результат (`rolled`),
  вообще без предсказуемого seed.

### H3. Инъекция в PostgREST-фильтр (`searchUsers`)
> ✅ **Исправлено** (2026-07-03): ввод санитизируется белым списком `[\p{L}\p{N} _-]` (режет `, ( ) . * %`), wildcard переведён на `*`. Клиентская правка (`src/lib/friends.ts`). Примечание: `profiles` и так публично-читаема по RLS, поэтому глубже (RPC) не требуется — данные и без того доступны.
```
.or(`username.ilike.${term}%,display_name.ilike.%${term}%`)
```
`term` подставляется в строку `.or()` без экранирования. Спецсимволы
(`,` `)` `(` `%` `_`) могут сломать/изменить фильтр (например `,` добавит новое
условие OR). SELECT на `profiles` и так публичный, но это некорректно и может
приводить к неожиданным результатам/ошибкам.
**Фикс:** валидировать `term` (разрешить буквы/цифры/`_`/пробел), экранировать
`%` и `_`, либо строить фильтр параметрически (`.ilike('username', pattern)` +
`.or` из заранее собранных безопасных кусков).

## 4. Проверка процедуры игры с ботом (прогон)

Мысленный прогон цикла `PlayPage → BotGameSession → useGame`:
1. Вход на `/play`: `setupOpen=true` → показывается ТОЛЬКО модалка настроек
   (доска не грузится) — корректно.
2. «Начать игру» → `start()` → `startTick+1` → `game.reset()` → фаза
   `openingRoll`.
3. Вступительный бросок: две кости, при равенстве переброс, определяется первый
   ход — корректно.
4. Ход человека: бросок (сервер не участвует, локально), выбор шашки → цели по
   `allowedMoves` (максимальность + больший кубик) → применение; при отсутствии
   ходов авто-пас → ход бота.
5. Ход бота: `chooseSequence` перебирает максимальные последовательности и
   выбирает по `evaluate` (pip, вынос, бар, блоты/пункты) — корректно для
   коротких.
6. Победа: `off==15` → фаза `gameover`, модалка «Победа/Поражение».
7. Пауза (открыт оверлей) корректно замораживает асинхронные шаги бота и кольцо
   таймера; после закрытия — продолжение. Снимок пишется в `sessionStorage`.

Ошибок в основном потоке не выявлено. Замечания — по длинным нардам (M5) и
недетерминизму восстановления (L4).
