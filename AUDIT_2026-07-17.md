## 0. Синхронизация с репозиторием (главный вопрос)

**Вывод: незакоммиченного кода, который можно потерять, НЕТ.**

- Локальная ветка `main` = `origin/main`, коммит `49729cb`. Расхождение
  **0 ahead / 0 behind** — локальная версия полностью совпадает с
  https://github.com/Mincorey/a-nards по коммитам.
- `git status` показывает 56 «изменённых» файлов, НО это **ложные изменения**:
  `git diff --ignore-all-space` пуст — единственное отличие — окончания строк
  **LF → CRLF** (в diff видны маркеры `^M`). Реального изменения кода нет ни в
  одном файле.
- **Причина:** в репозитории нет `.gitattributes`, а `core.autocrlf` не задан.
  Редактор/инструмент на Windows переписал файлы с CRLF, и git считает их
  изменёнными.
- **Единственный по-настоящему неотслеживаемый объект** — папка `_mont/`
  (3 PNG-монтажа скриншотов, ~6.5 МБ). Всё остальное (`_migration_fr/`,
  `vite.config.*.timestamp-*.mjs`, `*.tsbuildinfo`, `Supabase.txt`) уже корректно
  закрыто в `.gitignore`.

### Что сделать (безопасно, рекомендуется)
1. 🟠 ✅ **ВЫПОЛНЕНО (2026-07-17).** Добавить `.gitattributes` в корень и провести
   ренормализацию, чтобы прекратить «фантомные» CRLF-диффы навсегда:
   ```
   * text=auto eol=lf
   *.png binary
   *.jpg binary
   ```
   затем однократно: `git add --renormalize .` и один коммит.
   Это уберёт все 56 ложных изменений и предотвратит их повторное появление.
   > Реализация: создан `.gitattributes` (расширен: шрифты/аудио тоже помечены
   > `binary`), выполнен `git add --renormalize .`, сделан коммит `e8df68a`
   > (1 файл, только `.gitattributes` — без мусорного коммита 56 файлов).
   > `git status` чист, `git diff` пуст. Коммит **пока не запушен** — ветка
   > `main` на 1 впереди `origin/main`; запушить с рабочей машины: `git push`.
2. 🟢 Добавить `_mont/` в `.gitignore` (это рабочие скриншоты, не исходники) —
   либо удалить папку, если она больше не нужна.
3. 🟢 **Не коммитить** текущие 56 файлов «как есть» — иначе в истории появится
   мусорный коммит смены окончаний строк без смысловых правок.

> Примечание про упомянутые «в прошлой сессии незакоммиченные изменения»: это
> и есть та самая CRLF-разница. Никакая логика/фичи не потеряны.

## 2. Что сделано хорошо (не сломать при правках)

- **Сервер авторитетен.** Ходы онлайн-партии идут только через Edge Functions
  (`play-move`, `roll-dice`, `start-game`, `pass-turn`, `resign`,
  `claim-timeout`), которые заново валидируют легальность тем же движком. Клиент
  не пишет в `games`/`moves` напрямую.
- **Оптимистичная блокировка** в `play-move`: апдейт `games` привязан к
  `updated_at` (`.eq('updated_at', rev)`) — двойная отправка/гонка не применит
  ход дважды (409).
- **`finalize_game` идемпотентна** (миграция 0018): статистика и Elo-рейтинг
  начисляются ровно один раз — по факту смены статуса на `finished`
  (`get diagnostics row_count`). Гонка resign + claim-timeout безопасна.
- **Криптостойкие кубики** (`secureRng`), колонки `seed` в `games` нет — броски
  непредсказуемы (исправление из прошлого аудита на месте).
- **RLS продумана.** Пароль приватного стола — в отдельной таблице
  `table_secrets` без SELECT-политик; вход только через `join_table_secure`
  (`SECURITY DEFINER`, проверяет `auth.uid()`, число мест, пароль). Секреты
  контактов — `contact_messages` без политик, доступ только у service_role.
- **Realtime двухканальный:** быстрый путь через broadcast (`createGameSync`,
  <100 мс) + надёжный fallback `postgres_changes`, дедуп по `updated_at`.
  Чат — эфемерный broadcast, в БД не пишется.

## 3. База данных — находки advisors (боевой проект)

### 3.1 🟠 ✅ ВЫПОЛНЕНО (2026-07-17) Производительность RLS: `auth_rls_initplan`
Политики на `profiles`, `game_tables`, `table_seats`, `invites`, `friendships`,
`table_secrets` вызывают `auth.uid()` **на каждую строку**. При росте объёмов это
деградирует запросы. Фикс — обернуть в подзапрос, чтобы значение вычислялось один
раз: заменить `auth.uid()` → `(select auth.uid())` во всех политиках.
Рекомендуется отдельная миграция `0021_rls_initplan_optim.sql`. Эффект: заметно
дешевле SELECT/UPDATE на больших выборках (лобби, рейтинги, друзья).
Док: https://supabase.com/docs/guides/database/postgres/row-level-security#call-functions-with-select

> **Реализация:** создана миграция `supabase/migrations/0021_rls_initplan_optim.sql`
> (17 политик на 6 таблицах: `profiles`, `game_tables`, `table_seats`, `invites`,
> `friendships`, `table_secrets` — все прямые `auth.uid()` обёрнуты в
> `(select auth.uid())`; политики через `private.can_see_table(...)` не тронуты).
> Порядок: сухой прогон в `begin…rollback` (0 ошибок) → `apply_migration` на боевой
> проект `gmpvcrsmkxpjkjrvuyng`. Проверка после применения: `pg_policies` — 17
> обёрнуто / 0 необёрнуто; тексты политик идентичны оригиналам (изменилась только
> обёртка вызова, семантика сохранена); performance-advisor — **все 17
> предупреждений `auth_rls_initplan` исчезли**. Локальный SQL-файл миграции нужно
> закоммитить и запушить вместе с остальными изменениями.

### 3.2 🟡 ✅ ВЫПОЛНЕНО (2026-07-17) Непокрытые индексами FK: `unindexed_foreign_keys`
Отсутствуют covering-индексы у FK:
- `friendships.requester_id`, `friendships.addressee_id`
- `game_tables.owner_id`
- `invites.from_id`, `invites.to_id`
- `moves.player_id`
- `table_seats.user_id`

Влияние: медленные JOIN/фильтры и каскадные удаления при росте таблиц (особенно
`moves` — журнал ходов растёт быстрее всех). Фикс — миграция с
`create index if not exists ... on <table> (<fk_col>)` по каждому. Сейчас таблицы
почти пусты, так что применить дёшево и заранее.

> **Реализация:** создана миграция `supabase/migrations/0022_fk_covering_indexes.sql`
> — 7 индексов (`idx_friendships_requester/_addressee`, `idx_game_tables_owner`,
> `idx_invites_from/_to`, `idx_moves_player`, `idx_table_seats_user`). Применена через
> `apply_migration` к `gmpvcrsmkxpjkjrvuyng`. Проверка: все 7 индексов присутствуют
> в `pg_indexes`; performance-advisor — **все 7 предупреждений `unindexed_foreign_keys`
> исчезли**. Примечание: advisor теперь помечает эти 7 индексов как `unused_index` —
> это ожидаемо (таблицы пусты, трафика ещё не было), удалять НЕ нужно (та же природа,
> что у пункта 3.3). Локальный SQL-файл нужно закоммитить и запушить.

### 3.3 🟢 Неиспользуемые индексы: `unused_index` (не трогать)
`idx_game_tables_quick_waiting`, `idx_contact_ip_time`, `idx_contact_time`
помечены как неиспользуемые — **только потому что таблицы пустые** (0 строк) и

### 3.4 🟡 Безопасность (INFO/WARN — в основном by design)

  только у service_role через Edge `send-contact`). Ложное срабатывание, оставить.
- `join_table_secure` — `SECURITY DEFINER`, исполняема `authenticated` через
  `/rpc/`. Это **необходимо** для посадки за стол; функция сама проверяет
  `auth.uid()`, статус стола, места и пароль. Оставить, задокументировать.
- ~~🟠 **`auth_leaked_password_protection` выключена**~~ — ❌ ОТМЕНЕНО (2026-07-17):
  функция недоступна на бесплатном тарифе Supabase (Leaked Password Protection /

## 4. Финансовая часть (заглушки — оставлены без изменений)

Как и просили — ничего не трогал. Фиксирую текущее состояние и что понадобится
при подключении реальных денег.

**Текущее состояние — чистые заглушки, риска утечки денег нет:**
- `src/lib/wallet.ts`: `WALLET_BALANCE_RUB = 3500` — жёстко зашитый баланс,
  используется и в меню профиля, и на странице профиля (одно значение —
  совпадают). Реального баланса нет.
- В таблице `profiles` **нет колонок баланса/COINS** (есть только `rating`,
  `games_played`, `games_won`, `phone`). То есть кошелёк нигде не хранится.
- Ставка COINS у столов хранится в `game_tables.settings` (jsonb) —
  **справочная**, без резервирования и списания (`join_table_secure` COINS не
  трогает). Комментарий в миграции 0019 это подтверждает.
- Вывод средств: телефон сохраняется в `profiles.phone` (маска +7), кнопки
  вывода — заглушки.
- `LobbyPage`: `COIN_PRESETS = [50,100,150,200,500,1000]` — пресеты-заглушки.

**Что потребуется при переходе на реальную экономику (на будущее, НЕ сейчас):**
1. Таблица баланса/леджера: либо `profiles.coins int`, либо отдельная
   `wallet_ledger` (двойная запись: начисления/списания с историей).
2. Транзакционное **резервирование ставки** при посадке за стол за COINS
   (внутри `join_table_secure` или отдельной RPC, в одной транзакции с проверкой
   достаточности баланса).
3. **Расчёт по итогу** в `finalize_game`: перевод банка победителю, комиссия —
   строго идемпотентно (у `finalize_game` уже есть защита от повторного вызова —
   переиспользовать её `v_changed`-гейт).
4. Идемпотентность и аудит всех платёжных операций, серверная валидация сумм,
   защита от отрицательного баланса (`check`-констрейнт).
5. Платёжный провайдер для ввода/вывода рублей + вебхуки со сверкой.
