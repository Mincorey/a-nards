# Бот vs Онлайн: почему баги в режимах разные (аудит 2026-07-19)

Вопрос от тестирования: «игра с ботом и игра онлайн — это одна и та же игра,
но баги в режимах разные. Почему?»

## Короткий ответ

Общая у режимов только «нижняя половина»: движок правил (`src/engine/*`),
доска и вся отрисовка (`Board.tsx`, `Dice3D.tsx`, анимации, звуки). А вот
«верхняя половина» — управление ходом партии — реализована ДВУМЯ разными
машинами состояний:

| | Бот (`game/useGame.ts`) | Онлайн (`hooks/useOnlineGame.ts`) |
|---|---|---|
| Источник истины | локальная память (`gameRef`) | сервер (Supabase, Edge Functions) |
| Как приходят изменения | синхронно + `setTimeout`-хореография | оптимистично + 2 канала Realtime (broadcast и postgres_changes) |
| Фазы | `openingRoll → humanRoll → humanMove → botTurn → gameover` | `opening → myRoll → myMove → opponent → gameover` |
| rollId (триггер анимации костей) | счётчик `+1` за бросок | вычисляется из строки БД: `ply*2 + (rolled?1:0)` |
| Откат/повтор состояния | невозможен (один поток) | возможен (сеть: дубли, обгоны, откат при отказе сервера) |
| Таймер хода | только визуальное кольцо | кольцо + серверное присуждение поражения (claim-timeout) |

Поэтому баги и не совпадают: у бота ломается **хореография таймеров**
(лишние/двойные анимации из-за `sleep`/StrictMode — истории с двойным
кувырком костей, фантомом при выносе), а в онлайне ломается **синхронизация
состояний** (перетирание оптимистичного состояния строками из сети — свежий
пример: фантомный «второй бросок костей» при ходе цепочкой).

## Разбор известных расхождений

1. **Фантомный второй бросок костей (только онлайн).** Ход на самую дальнюю
   зелёную точку — это цепочка (оба кубика одной шашкой). Клиент оптимистично
   применял всю цепочку, а сервер исполнял её двумя UPDATE'ами; realtime-строка
   ПЕРВОГО полухода была «свежее» по `updated_at` и на миг откатывала UI назад
   → `rollId` менялся → `Dice3D` заново кувыркал кости. У бота такого класса
   багов нет в принципе — нет сети. **Исправлено:** входящие строки теперь
   сравниваются по логическому прогрессу партии (`game/gameOrder.ts` +
   тесты), строки «из прошлого» отбрасываются.

2. **Двойная анимация костей (только бот, исправлено ранее).** React
   StrictMode дважды прогонял эффект кувырка — у онлайна не проявлялось из-за
   иного жизненного цикла rollId.

3. **Модалка победы исчезала (оба режима, но по-разному).** Ghost-click после
   тапа — в боте проявлялся чаще, т.к. модалка монтировалась синхронно в
   обработчике. Исправлено «взводом» Modal + задержкой GameOverModal.

4. **Таймер хода.** У бота кольцо чисто визуальное (20с/45с), в онлайне — 90с
   основного + 30с дополнительного времени с присуждением поражения
   (см. `game/timeout.ts`, edge-функция `claim-timeout`). Это осознанное
   расхождение: боту спешить некуда.

## Как сблизить режимы (рекомендация на будущее)

Полное слияние — вынести общий «редьюсер партии» (фазы, выбор шашки, цепочки,
авто-пас) в один модуль, а различие свести к «драйверу соперника»:
локальный бот или сервер. Это отдельный рефакторинг (~1-2 сессии); текущие
точечные меры (общий движок, общая доска, `gameOrder.ts`, одинаковые тайминги
авто-паса 2400мс и задержки модалки 900мс) уже закрывают все известные
видимые расхождения.

## Чек-лист при новых «разных» багах

- Баг только в онлайне → ищи в синхронизации: `useOnlineGame.applyIncoming`,
  каналы в `lib/online.ts`, оптимистичные ветки `pick()`.
- Баг только с ботом → ищи в хореографии: `useGame.ts` (sleep/setTimeout,
  паузы, StrictMode-двойной прогон эффектов).
- Баг в обоих → движок/доска: `engine/*`, `Board.tsx`, `useBoardAnimations.ts`.
